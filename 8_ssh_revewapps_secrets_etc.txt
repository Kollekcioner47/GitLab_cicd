Практика 8: Полноцикловый проект с нуля — CRM, SSH-деплой, Review Apps и секреты

В этой практике мы создадим проект с нуля, чтобы освежить навыки работы с Git, и реализуем многоконтейнерное приложение: простую CRM-систему. 

Архитектура приложения:
1. PostgreSQL — база данных для хранения клиентов и сделок.
2. Backend (Бизнес-логика) — REST API на Python (FastAPI), которое работает с БД.
3. Frontend (CRM UI — Веб-интерфейс на Python, который отдаёт HTML и обращается к Backend.

Чему мы научимся:
- Создавать проект в GitLab с нуля.
- Настраивать деплой по SSH на удалённый сервер (10.0.2.6).
- Использовать параллельные тесты и генерировать JUnit-отчеты.
- Создавать временные окружения (Review Apps) для веток.
- Управлять секретами через CI/CD Variables и очищать артефакты.

**2. Backend (Бизнес-логика)**
Создайте `backend/app.py`:

import os
from fastapi import FastAPI, Depends
from sqlalchemy import create_engine, Column, Integer, String, Numeric
from sqlalchemy.orm import declarative_base, sessionmaker, Session
from pydantic import BaseModel

DATABASE_URL = os.getenv("DATABASE_URL", "postgresql://crm:secret@db:5432/crmdb")

engine = create_engine(DATABASE_URL)
SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)
Base = declarative_base()


class Deal(Base):
    __tablename__ = "deals"
    id = Column(Integer, primary_key=True, index=True)
    client_name = Column(String(100), nullable=False)
    amount = Column(Numeric(10, 2), nullable=False)
    status = Column(String(20), default="new")


Base.metadata.create_all(bind=engine)
app = FastAPI(title="CRM Business Logic")


class DealCreate(BaseModel):
    client_name: str
    amount: float


def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()


@app.get("/api/deals")
def list_deals(db: Session = Depends(get_db)):
    return db.query(Deal).all()


@app.post("/api/deals")
def create_deal(deal: DealCreate, db: Session = Depends(get_db)):
    db_deal = Deal(client_name=deal.client_name, amount=deal.amount)
    db.add(db_deal)
    db.commit()
    db.refresh(db_deal)
    return db_deal

3. Frontend (CRM UI)
Создайте `frontend/app.py`:

import os
import httpx
from fastapi import FastAPI, Request
from fastapi.responses import HTMLResponse

API_URL = os.getenv("API_URL", "http://backend:8000")

app = FastAPI()


@app.get("/", response_class=HTMLResponse)
async def index(request: Request):
    async with httpx.AsyncClient() as client:
        resp = await client.get(f"{API_URL}/api/deals")
        deals = resp.json()
    
    rows = "".join(
        f"<tr><td>{d['id']}</td><td>{d['client_name']}</td>"
        f"<td>{d['amount']}</td><td>{d['status']}</td></tr>"
        for d in deals
    )
    
    return f"""
    <html><head><title>Mini CRM</title></head><body>
    <h1>CRM Deals</h1>
    <table border="1" cellpadding="5">
    <tr><th>ID</th><th>Client</th><th>Amount</th><th>Status</th></tr>
    {rows}
    </table></body></html>
    """

Шаг 7. Работаем с ветками и Review Apps

Настало время проверить магию Review Apps. Разработчик хочет добавить новую фичу.

Создадим ветку:

git checkout -b feature/add-status


Изменим `frontend/app.py`, чтобы статус подсвечивался цветом. Найдите строку с `{d['status']}` и замените на:

f"<td style='color:green;'>{d['status']}</td></tr>"

То есть в итоге должно получиться:
rows = "".join(
    f"<tr><td>{d['id']}</td><td>{d['client_name']}</td>"
    f"<td>{d['amount']}</td><td style='color:green;'>{d['status']}</td></tr>"
    for d in deals
)

Отправим ветку:

git add .
git commit -m "Highlight deal status in green"
git push origin feature/add-status


Теперь в GitLab создайте **Merge Request** из `feature/add-status` в `main`.
Как только вы создадите MR, запустится пайплайн, который:
1. Прогонит параллельные тесты.
2. Соберёт новые образы.
3. Задеплоит ваше изменение как **Review App** на порт `808ХХ` (где ХХ — последние символы хеша коммита) на том же сервере 10.0.2.6!

Во вкладке **Environments** в GitLab вы увидите новое окружение `review/1` (где 1 — номер MR), а в самом Merge Request появится кнопка **View app**, ведущая прямо на ваш изолированный стенд!

Шаг 8. Слияние и очистка

1. Нажмите **Merge** в Merge Request.
2. Код попадёт в `main`, запустится пайплайн `main`, который задеплоит Staging.
3. Окружение Review App автоматически не удалится, но вы можете перейти в **CI/CD -> Environments**, найти ваше `review/1` окружение и нажать кнопку со стопором (или использовать кнопку `Stop` в самом MR). При этом запустится джоба `stop_review`, которая по SSH зайдёт на 10.0.2.6 и выключит контейнеры этой ветки — **очистка ресурсов!**
4. Артефакты (отчеты JUnit) удалятся автоматически через 1 день (`expire_in: 1 day`), освобождая место на диске GitLab.

Шаг 9. Отправка уведомлений

Шаг 2. Python-скрипт для отправки письма
Создайте в корне проекта файл send_email.py:
#!/usr/bin/env python3
import os
import smtplib
import ssl
from email.message import EmailMessage
import sys

def send_email(subject, body, to_email):
    smtp_server = os.getenv('SMTP_SERVER')
    smtp_port = int(os.getenv('SMTP_PORT', '587'))
    smtp_user = os.getenv('SMTP_USER')
    smtp_password = os.getenv('SMTP_PASSWORD')
    from_email = smtp_user   # обычно совпадает с пользователем

    msg = EmailMessage()
    msg.set_content(body)
    msg['Subject'] = subject
    msg['From'] = from_email
    msg['To'] = to_email

    try:
        # Определяем контекст SSL для порта 465, иначе STARTTLS
        if smtp_port == 465:
            with smtplib.SMTP_SSL(smtp_server, smtp_port, context=ssl.create_default_context()) as server:
                server.login(smtp_user, smtp_password)
                server.send_message(msg)
        else:
            with smtplib.SMTP(smtp_server, smtp_port) as server:
                server.starttls(context=ssl.create_default_context())
                server.login(smtp_user, smtp_password)
                server.send_message(msg)
        print(f"Email sent to {to_email}")
        return True
    except Exception as e:
        print(f"Failed to send email: {e}", file=sys.stderr)
        return False

if __name__ == "__main__":
    # Аргументы командной строки: subject, body, to_email (опционально, берём из переменной)
    subject = sys.argv[1] if len(sys.argv) > 1 else "GitLab CI/CD Notification"
    body = sys.argv[2] if len(sys.argv) > 2 else "No details provided."
    to_email = sys.argv[3] if len(sys.argv) > 3 else os.getenv('ALERT_EMAIL_TO')
    if not to_email:
        print("No recipient email provided. Set ALERT_EMAIL_TO variable or pass as argument.", file=sys.stderr)
        sys.exit(1)
    success = send_email(subject, body, to_email)
    sys.exit(0 if success else 1)

Шаг 3. Добавляем джобу notify в .gitlab-ci.yml
Добавим новую стадию notify после deploy. Также добавим возможность отправлять уведомления при успехе и при падении (используя when: always).

Обновите .gitlab-ci.yml:
stages:
  - test
  - build
  - deploy
  - cleanup
  - notify          # новая стадия

# ... остальные джобы (test_backend, test_frontend, build_images, deploy_staging, deploy_review, stop_review, deploy_production)

# Джоба уведомления – запускается после deploy_staging и deploy_production (успешных или нет)
notify:
  stage: notify
  image: python:3.11-slim
  tags:
    - docker
  script:
    - pip install --no-cache-dir  # стандартная библиотека, ничего не ставим
    - |
      if [ "$CI_JOB_STATUS" == "success" ]; then
        STATUS_EMOJI="✅"
        SUBJECT_SUFFIX="SUCCESS"
      else
        STATUS_EMOJI="❌"
        SUBJECT_SUFFIX="FAILURE"
      fi
      SUBJECT="[${SUBJECT_SUFFIX}] ${CI_PROJECT_NAME} pipeline #${CI_PIPELINE_ID} (${CI_COMMIT_BRANCH})"
      BODY="Status: ${CI_JOB_STATUS} (${STATUS_EMOJI})
      Project: ${CI_PROJECT_URL}
      Branch: ${CI_COMMIT_BRANCH}
      Commit: ${CI_COMMIT_SHORT_SHA} - ${CI_COMMIT_TITLE}

      "
      python send_email.py "$SUBJECT" "$BODY"
  when: always                     # отправляем всегда, независимо от результата предыдущих джоб
  only:
    - main                         # или можно отправлять для всех веток
  except:
    - merge_requests               # не отправляем для MR, чтобы не дублировать (опционально)
  needs:
    - deploy_staging               # дожидаемся деплоя в staging (можно указать несколько)
  # Если хотите также уведомления после деплоя в production, раскомментируйте:
  #   - deploy_production
